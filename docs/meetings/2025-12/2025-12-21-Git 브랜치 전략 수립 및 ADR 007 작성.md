# 2025-12-21-Git 브랜치 전략 수립 및 ADR 007 작성

## Summary (회의 요약)
Git 워크플로우를 확정하고 ADR 007을 작성함. Modified GitHub Flow(develop 브랜치 추가), Issue 기반 개발 프로세스, Labels 체계, Epic Issue 전략을 수립하고 Phase별 즉시 배포 방식을 결정함.

## Key Decisions (주요 결정 사항)

### Modified GitHub Flow 브랜치 전략 채택
- **내용:** `main`(production), `develop`(통합), `feature/#N-기능명` 구조 사용. 브랜치 네이밍은 하이픈(`-`) 사용.
- **이유:** 회사 워크플로우와 일관성 유지하며 실무 경험 축적. Git Flow보다 단순하고 순수 GitHub Flow보다 통합 테스트 완충지대 확보 가능.

### Issue/PR 규칙 확정
- **내용:** Issue 제목 `Backend Hello World 구현`, PR 제목 `Backend Hello World 구현 (#15)` 형식. Phase는 제목에 포함하지 않고 Labels/Milestones로 관리.
- **이유:** 제목만으로도 컨텍스트 명확하고 GitHub 검색 친화적. 이슈 번호는 뒤에 배치하여 가독성 향상.

### GitHub Labels 체계
- **내용:** Type(`feature`, `bug`, `documentation`), Area(`backend`, `frontend`, `infrastructure`), Phase(`phase-1`, `phase-2`, `phase-3`), Priority(`priority-p0`, `priority-p1`, `priority-p2`) 정의.
- **이유:** 필터링/통계 활용 및 우선순위 시각화. 일반 기능은 Priority 라벨 없이 진행.

### Epic Issue + Sub-issues 전략
- **내용:** 큰 작업은 Epic Issue로 관리하고 Sub-issue들을 생성하되, 모든 브랜치는 `develop`에서 직접 분기. Sub-feature 브랜치는 생성하지 않음.
- **이유:** Short-lived Branches로 Merge Conflict 최소화, 병렬 작업 가능, Rollback 용이.

### Phase별 즉시 배포 전략
- **내용:** Phase 하나씩 완료할 때마다 `develop` → `main` 병합하여 AWS EC2에 배포. Semantic Versioning(v0.1.0, v0.2.0, v1.0.0) 적용.
- **이유:** Walking Skeleton 원칙에 따라 점진적 배포 경험 축적 및 리스크 분산.

### AI 코드 리뷰 활용
- **내용:** Self-Review의 한계를 보완하기 위해 GitHub Copilot, ChatGPT 등으로 PR 리뷰 수행.
- **이유:** 1인 프로젝트에서 코드 품질 유지 및 포트폴리오 차별화.

## TIL (느낀 점)
- 회사 워크플로우를 개인 프로젝트에 적용하면 습관화되어 실무 적응력 향상.
- 브랜치 네이밍(하이픈 vs 언더스코어)처럼 작은 선택도 커뮤니티 표준을 따르기로 함
- Epic Issue로 큰 작업을 쪼개되 브랜치는 모두 develop에서 직접 분기하는 방식이 1인 프로젝트에 최적임.
