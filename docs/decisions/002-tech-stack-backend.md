# 2. Backend Tech Stack Selection

**Status:** Accepted
**Date:** 2025-12-14
**Last Updated:** 2025-12-15

## Context (배경 및 문제점)
- SpaceSignal의 백엔드는 위성 궤도 데이터(TLE)를 실시간으로 처리하고 계산해야 함.
- 초기 개발 속도가 빨라야 하며, 추후 고성능 모듈(Go) 도입을 고려한 유연한 아키텍처가 필요함.
- 위성 궤도 역학 관련 라이브러리 지원이 필수적임.

## Decision (결정 사항)
백엔드 핵심 기술로 **Python 3.13+**와 **FastAPI** 프레임워크를 선정함.

1. **Python 3.13:** 최신 안정 버전(Stable)을 사용하여 성능 개선 및 장기 지원(LTS) 이점을 확보.
2. **FastAPI:** 비동기(Async) 처리에 최적화되어 있으며, 타입 힌트 기반의 개발 경험이 우수함.
3. **Skyfield:** Python 생태계에서 가장 정밀하고 검증된 위성 궤도 계산 라이브러리 채택.

## Consequences (결과 및 효과)
### Positive (장점)
- **라이브러리 생태계:** Skyfield, Pandas 등 과학 계산용 라이브러리가 풍부하여 복잡한 수식을 직접 구현할 필요 없음.
- **개발 생산성:** FastAPI의 자동 문서화(Swagger UI) 기능 덕분에 프론트엔드와의 협업 속도가 빠름.
- **확장성:** 마이크로서비스 구조(MSA)로 전환하거나 일부 API를 Go 언어로 분리하기 좋은 가벼운 구조임.

### Negative (단점/리스크)
- **실행 속도:** C++이나 Go에 비해 연산 속도가 느릴 수 있으나, Phase 1~2 단계의 트래픽에서는 충분함. (추후 Go 도입으로 해결 예정)

## Alternatives (대안)
- **Django:** 기능은 강력하나, 단순 API 서버로는 너무 무겁고 비동기 처리가 상대적으로 복잡함.
- **Node.js:** 웹 개발엔 좋으나, Skyfield 수준의 정밀한 위성 궤도 계산 라이브러리가 부족함.
